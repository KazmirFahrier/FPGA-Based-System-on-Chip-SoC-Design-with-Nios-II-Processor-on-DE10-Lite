01 / Overview

Why build the chip yourself?

Off-the-shelf microcontrollers solve common problems — but they freeze the architecture in silicon. This project takes the opposite approach: synthesize the processor, the memory, the interconnect, and the I/O peripherals into the FPGA fabric itself, so the entire system is a configuration choice. The result is a teaching-grade demonstration of the full hardware/software flow that production embedded engineers run every day.

Developed as part of the University of Colorado Boulder FPGA Design for Embedded Systems capstone — extended here with reproducible documentation and lab artifacts.

  • Configurable, not fixed. Build a soft SoC instead of relying on a hard-processor MCU.
  • Real interconnect. Wire processor, memory, clocks, debug, and I/O via Avalon-MM fabric.
  • Working firmware. Author a Nios II C application that exercises the generated hardware.
  • Hardware validation. Verify on real silicon — LEDs, switches, timers, JTAG UART output.
  • Reproducible artifacts. Preserve the capstone guides and lab notebook so the work can be rebuilt.
02 / Architecture

The SoC, diagrammed.

The Nios II Gen2 processor sits at the center, with instruction and data masters reaching memory and peripherals through Avalon-MM interconnect fabric. A clock-crossing bridge isolates the high-speed CPU/SDRAM domain from the slower peripheral domain. Two PLLs synthesize the clocks; a SysID peripheral guarantees firmware and gateware never drift out of sync.

03 / IP Inventory

Eleven blocks, one integrated system.

Every Platform Designer block was selected and wired deliberately — clocks routed to the right domains, masters paired with the right slaves, interrupts and reset signals connected end-to-end.

IP · 01CPU
Nios II Gen2 nios2_gen2_0
80 MHz soft processor with separate instruction and data masters. The nucleus of the SoC.
IP · 02MEMORY
On-Chip RAM onchip_ram
16 KB of fabric memory mapped as the reset and exception target for the Nios II core.
IP · 03MEMORY
On-Chip Flash onchip_flash_0
MAX 10 internal flash region, accessible to the processor for non-volatile storage.
IP · 04MEMORY
SDRAM Controller sdram
External SDRAM controller for the DE10-Lite — bulk working memory beyond on-chip resources.
IP · 05I/O
JTAG UART jtag_uart
Console I/O channel between the Nios II program and the host computer over JTAG.
IP · 06TIMING
Avalon Timer timer_0
32-bit Avalon timer driving HAL alarms and firmware-side delay routines (1 / 10 / 100 / 250 ms).
IP · 07I/O
LED PIO led_pio
10-bit output PIO mapped directly to the DE10-Lite red LEDs for the count display.
IP · 08I/O
Slide PIO slide_pio
10-bit input PIO on the slide switches with edge-interrupt support for runtime user input.
IP · 09I/O
SPI Master spi_0
SPI controller configured for the on-board accelerometer / G-sensor interface.
IP · 10ANALOG
Modular ADC modular_adc_0
MAX 10 modular ADC block with channel 1 enabled — sample store and sequencer CSRs exposed.
IP · 11SAFETY
System ID sysid
Validates at runtime that the firmware was built against the same gateware now loaded on the FPGA.
·FABRIC
Avalon-MM Interconnect
Generated by Platform Designer. Routes masters to slaves with a clock-crossing bridge between domains.

06 / Build & Run

From RTL to LEDs, in nine steps.

A reproducible flow from a fresh checkout to blinking lights — the same sequence used during the capstone validation.

Open the Quartus project

Launch hardware/Embed.qpf in Quartus Prime Lite. Confirm the device is recognized as 10M50DAF484C6GES.

Quartus Prime Lite

Regenerate the SoC

Open hardware/Embed.qsys in Platform Designer. Regenerate HDL if any blocks have changed. Inspect the interconnect for missing IP warnings.

Platform Designer

Compile the gateware

Run a full Quartus compile to produce the FPGA programming file. Keep an eye on timing closure if PLL or memory parameters were edited.

Quartus · Compile

Program the DE10-Lite

Use the Quartus Programmer over USB Blaster to load the .sof onto the MAX 10.

USB Blaster · JTAG

Generate the BSP

Create or refresh the Nios II Board Support Package from the generated hardware description. The BSP produces a system.h matching the gateware.

Nios II SBT

Pull in the firmware

Add the files in software/ to the Nios II application project. Confirm the helper modules (alarm, delay, JTAG UART, LED) compile without warnings.

Eclipse / SBT

Build the application

Compile the firmware with the Nios II Software Build Tools — either the Eclipse front end or the command-line SBT flow.

GCC for Nios II

Run on target

Launch the ELF onto the active Nios II target and open the Nios II console window connected to the JTAG UART.

JTAG UART Console

Validate on hardware

Type u, d, or 3 in the console — and watch the red LEDs march through the count sequence on the board.

Bench validation ✓
Validation checklist
  • Quartus recognizes the device as 10M50DAF484C6GES.
  • Platform Designer generates the Embed system without missing IP warnings.
  • BSP system.h defines LED_PIO_BASE, JTAG_UART_BASE, TIMER_0_BASE, SYSID_BASE, and the Nios II CPU.
  • Software prints the startup banner in the JTAG UART console.
  • LEDs update in the same sequence printed in the console.
  • Invalid console input is handled without crashing the application.
